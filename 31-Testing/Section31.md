# 테스팅 (Testing)

### 테스팅이란 무엇이며 왜 해야할까?

- 내가 작성한 코드의 애플리케이션이 잘 작동하는지 확인필요
- 수동 테스트 보단 자동화된 테스팅으로도 애플리케이션 테스트가 가능함 => 자동화된 테스팅
- 수동 테스트를 진행할 경우, 많은 시간이 소요되므로 자동화된 테스팅 이용시 시간절약가능

### 테스트의 장점

- 코드의 큰 변경 사항이나 문제를 볼 수 있고 애플리케이션의 전혀 다른 곳에서 변경되면서 발생한 문제도 즉시 확인가능함

### 효율적인 테스트 방법

- 복잡한 의존성 분해 -> 테스팅, 코드 작업 및 개선 용이해짐

### 테스트 유형

1. 유닛 테스트

- 애플리케이션의 분리된 단위로 하나를 테스트 (분리된 코드 조각)
- 단위가 작고 결과에 대한 추측이 쉬움

2. 통합 테스트

- 의존성을 가진 단위에 대한 테스트

3. 사용자 인터페이스 테스트 || E2E(End-to-End) 테스트

- 전체 애플리케이션 (일부) 테스트

---

## 테스트시 필요한 도구

### 1. 테스트 러너

- 테스트를 실행하고 결과를 요약해서 결과에 대한 출력을 제공함
- 예: Mocha, Jest
- 유닛 및 통합 테스트시 사용 (E2E 테스트도 가능)

### 2. 어서션(Assertion) 라이브러리

- 테스트 논리와 예상 결과를 정의함
- 도구를 제공하여 예상 및 테스트의 일부로써 확인하고 싶은 비교 및 조건 등을 정의함
- 예: Chai, Jest
- 유닛 및 통합 테스트시 사용 (E2E 테스트도 가능)

### 3. 헤드리스 브라우저

- E2E 테스트시 필요
- 수동으로 클릭하지 않아도 되고. 브라우저 API와 DOM 등을 사용할 수 있음
- 코드에서 분석하기 때문에 사용자 인터페이스는 필요치 않음
- 예: Puppeteer
- 유닛 및 통합 테스트에서는 사용불가
